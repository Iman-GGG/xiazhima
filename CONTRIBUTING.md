# 参与贡献

感谢你关注瞎芝麻。我们欢迎可复现、可审查并尊重项目风险边界的贡献。

## 可以贡献什么

- 修复明确、可复现的 Bug；
- 补充单元测试、类型检查和自动化验证；
- 改善文档、无障碍体验、响应式布局或错误提示；
- 优化行情解析、缓存、并发控制和预计算流程；
- 在具有明确规则来源、样本和测试的前提下改进量化规则实现。

涉及管理员鉴权、部署安全、数据源替换或核心信号规则的改动，请先创建 Issue 讨论。

## 开始之前

1. 搜索现有 Issue 和 Pull Request，避免重复工作。
2. Bug 请提供复现步骤、预期行为、实际行为和运行环境。
3. 较大的功能或规则变更请先创建 Issue，说明范围和验证方案。
4. 不要在 Issue、日志、截图或提交中公开密码、Cookie、令牌、环境变量值或用户数据。

## 本地开发

环境要求：Node.js 20+、pnpm 9+。本项目只使用 pnpm。

```bash
git clone https://github.com/Iman-GGG/xiazhima.git
cd xiazhima
pnpm install
pnpm dev
```

从最新的 `main` 创建主题分支：

```bash
git switch main
git pull --ff-only
git switch -c fix/short-description
```

推荐的分支前缀：`fix/`、`feat/`、`docs/`、`test/`、`refactor/`、`security/`。

## 提交前检查

根据改动范围运行以下命令：

```bash
pnpm test
pnpm ts-check
pnpm lint:build
pnpm build
```

至少确保与你的改动直接相关的测试通过。无法运行某项检查时，请在 Pull Request 中解释原因。

## 规则变更要求

修改 `src/lib/stock/b1.ts`、`indicators.ts` 或相关阈值时，Pull Request 必须包含：

1. 规则来源、推导依据或问题背景；
2. 可公开且可复现的输入样本；
3. 正向、反向和边界测试；
4. 对 B1、B2、单针下三十、S1 或大势判定的影响；
5. 是否需要同步修改战法学堂和风险提示。

不要只修改前端文案来改变裁断结论。业务规则应集中在 `src/lib/stock/` 中，界面只负责展示计算结果。

## 数据与安全要求

- 不提交 `.env`、真实管理员凭据、会话密钥或部署令牌；
- 不提交包含个人信息或真实交易记录的样本；
- 新增第三方数据源时，说明来源、用途、失败策略和适用条款；
- 外部输入必须经过格式与范围校验；
- 管理接口应保持默认拒绝、最小权限和安全 Cookie 策略；
- 安全漏洞请按照 `SECURITY.md` 私下报告，不要先创建公开 Issue。

## Pull Request 说明

请保持每个 Pull Request 目标单一，并说明：

- 解决的问题；
- 主要改动；
- 验证命令和结果；
- UI 改动前后的截图（如适用）；
- 风险、兼容性和后续工作；
- 关联 Issue，例如 `Closes #123`。

维护者可能要求补充测试、拆分改动或调整实现。提交代码即表示你有权贡献相关内容，并同意该贡献按项目的 Apache License 2.0 发布。

## 行为准则

请保持专业、友善和聚焦事实。对规则、数据和实现的分歧应通过可复现证据与技术讨论解决，避免人身攻击、骚扰或歧视性表达。
